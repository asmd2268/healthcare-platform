# التدقيق والتراجع الآمن

كل تغيير مؤثر يسجل: السجل والوحدة والمنشأة، الفاعل، الوقت، الإجراء، الحالة السابقة واللاحقة أو ملخص فرق آمن عندما يسمح، ومصدر العملية، ومعلومة الجهاز أو الجلسة عند الملاءمة. سجل التدقيق ليس بديلًا عن سجل النشاط التشغيلي، ولا يعدله المستخدم العادي.

CAPA يسجل الإنشاء والربط والإرسال والاعتماد والرفض والتعيين وبدء/إكمال/تحقق action والتمديد والفاعلية وإعادة الفتح والإغلاق والإلغاء والأرشفة والاستعادة والأدلة والتصدير. فشل module event أو shared audit يلغي العملية الخاضعة للمعاملة، ولا يسمح المستخدم العادي بإدراج `capa_events` مباشرة.

تتيح المنصة صفحة موحدة بعنوان «آخر التغييرات خلال 24 ساعة / Changes in the Last 24 Hours» للمستخدمين المصرح لهم. التراجع يعكس تغييرًا محددًا ضمن الصلاحية فقط، وينشئ حدث تدقيق جديدًا؛ لا يمحو الاعتمادات أو الصادرات أو الأدلة، ولا يتجاوز تعديلات لاحقة أو تعارضات. لا تُتراجع السجلات المعتمدة أو الموقعة أو المنشورة أو الحساسة قانونيًا بصمت. بعد 24 ساعة يستخدم مسار إعادة فتح/تصحيح رسمي مع سبب وموافقة عند اللزوم.

تتضمن عقود الإدارة أحداث form version creation وsoft delete وrestore request وpermanent deletion request.

## الأساس المنفذ

جدول `audit_events` append-only للمستخدمين العاديين ومقيد بـRLS، مع دالة خادمية `append_audit_event` ترفض كلمات المرور وtokens والأسرار في metadata. يغطي المخطط أحداث role assignment وpermission وform وreference data وimport/export وdeletion. واجهة Last 24 Hours تبقى محدودة حتى توفير tenant context وصلاحية فعلية للمستخدم.

التراجع التشغيلي غير منفذ بعد؛ الاستعادة للحذف المنطقي وطلبات الحذف فقط هي الأساس الحالي، ولا يوجد حذف غير قابل للعكس للسجلات المحمية.

محرك Workflow يضيف عقود events موثوقة للانتقال والتعيين والاعتماد وSLA والتصعيد والمهام. لا يسمح للمستخدم العادي بإنشاء system event مباشرة؛ التدوين التشغيلي الكامل يبقى مرتبطًا بدوال خادمية مقيدة عند تفعيل runtime.

## السياسات والإجراءات

أحداث policy domain append-only: الإنشاء، إنشاء النسخة، تعديل metadata، الإرسال، الاعتماد، الرفض، النشر، الاستبدال، الأرشفة، الاستعادة، رفع/تنزيل المستند، والإقرار. لا توجد سياسة RLS تسمح بالـinsert/update/delete المباشر للمستخدم العادي على `policy_events`. تستدعي دوال lifecycle المقيدة كاتبًا داخليًا وتسجل أيضًا Audit Log مشتركًا ضمن المعاملة. لا يوجد Safe Undo للنشر أو حذف مستند منشور؛ التصحيح بإصدار جديد.

يسجل مسار الرفع الموثوق إنشاء metadata وحذف مسودة فقط بعد التحقق الخادمي. لا يسجل أو ينفذ browser finalization، ولا يحذف orphan cleanup ملفًا مؤرشفًا أو منشورًا أو مرتبطًا بسجل نهائي.

تستخدم عمليات Storage التي تعمل بـservice role مسار تدقيق داخليًا يقبل actor تم التحقق من عضويته وصلاحيته، وليس `auth.uid()` الخاص بالـworker. يسجل المسار actor نفسه في سجل السياسة وAudit Log المشترك، ولا يمنح public أو anon أو authenticated تنفيذ هذه الدوال. أي فشل في insert التدقيق يفشل المعاملة كاملة.

## هويات الأتمتة الموثوقة

`service_role` هو سياق تنفيذ موثوق، وليس هوية تدقيق قابلة للإسناد. تسجل العمليات الآلية هوية مستقلة من `automation_identities` مرتبطة بـ`user_profiles` حتى تبقى مفاتيح `actor_id` الحالية وسجل التدقيق الموحد صالحين. لا تقبل الدوال الآلية UUID بشريًا يختاره مستدعي service role: يجب أن تكون الهوية نشطة، مسجلة لغرض صريح، ومطابقة لنطاق tenant/organization/facility. إلغاء التفعيل يحفظ الإسناد التاريخي ولا يحذف السجل؛ الدوران ينشئ principal جديدًا بدل إعادة توظيف الهوية القديمة.

المطبق حاليًا هو الغرض `inventory.reservation_expiry`. تسجل أوامر انتهاء حجز المخزون، وتعديلات `expiry_released`، وأحداث التحويل، و`audit_events` principal الأتمتة نفسه. لا توجد هوية بشرية بديلة في هذا المسار المجدول. عند إضافة عملية «نيابة عن» مستقبلًا، تسجل هوية الأتمتة والطالب البشري المتحقق منه في حقول/metadata منفصلة وآمنة، ولا تستبدل أحدهما بالآخر.

تسجل المرحلة الأولى لحذف مسودة policy document طلب الحذف، وتسجل المرحلة الثانية الحذف النهائي فقط بعد نجاح Storage. يحتفظ فشل Storage بالـmetadata والطلب كحالة قابلة للاسترداد؛ لا تستخدم عملية cleanup كـSafe Undo ولا تحذف ملفات retained أو finalized.
